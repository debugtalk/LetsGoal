# DataClaw 产品规划

## 1. 背景 & 痛点

### 1.1 现状痛点

评测技术团队早已实现了丰富的自动化采集能力，定时任务理论上可以自动跑。但实际情况是：脚本跑归跑，人还是离不开——检查执行、核验结果、处理异常、归因分析、临时补采，大量精力消耗在"自动化运维"上。

具体痛点：

- **执行过程离不开人**。定时任务跑没跑、哪个平台成功了、哪个失败了，全靠人工逐个确认；遇到页面改版、接口变化、风控拦截、设备离线等问题，也需要人工介入排查修复。
- **结果判断离不开人**。采集结果是否可用依赖人工判断（条数够不够、字段全不全、数据是否置信）；发现数据缺失要手动补采，发现指标异常要人工归因排查是脚本问题、环境问题还是数据源本身变化。
- **进展感知靠人主动**。要么完全不通知，人只能主动巡检；要么通知泛滥无重点，人还是要手动翻看原始结果才能知道真正发生了什么。无论哪种，最终都靠人自己去"看"。
### 1.2 问题本质

自动化只是起点，不是终点。

当前的状态是"**半自动化陷阱**"——脚本解决了执行环节，但任务管理、进展跟踪、异常处理、结果验收、归因分析、补采闭环仍然高度依赖人工。人没有被解放出来，只是从"手动采集"变成了"自动化运维"。

真正需要的是一位**靠谱的数字员工**：把采集任务委托给它，它自行保障执行、自行验收结果、自行处理常规异常、自行补采修复，只在搞不定时才向上汇报求助。用户从"盯过程"变成"关注关键结果、进展和风险"。

## 2. 目标 & 愿景

### 2.1 产品愿景

DataClaw 是评测团队的数字化采集员工。用户只需一次性配置采集目标与验收标准，DataClaw 即可自主完成执行跟踪、结果验收、异常自愈和主动汇报；常规问题自行解决并持续优化 Skills，仅在无法处理时请求人工介入。

### 2.2 核心价值主张

1. **从"人盯自动化"到"自动化自治"**：任务只配一次，DataClaw 自行跟踪执行、验收结果、触发补采，用户不再需要逐任务人工确认
2. **异常自愈优先，人工兜底**：基于沉淀的 Skills 自动处理页面变化、接口波动等常规异常；未知问题自动探索尝试，搞不定才通知人工，并附带已尝试的方案和失败原因
3. **全程主动汇报，关键信息直达**：每次执行完成后推送结果摘要（成功率、数据量、关键指标）；遇到风险即时告警，用户从"主动巡检"变为"按需接收"
4. **能力持续进化**：高频异常的处理经验自动沉淀为 Skill 优化建议，自定义任务的通用逻辑可一键复用为新 Skill，系统越用越稳

## 3. 建设思路

### 3.1 统一闭环范式

所有采集场景可抽象为统一的五步闭环：

```
Plan（规划）→ Execute（执行）→ Evaluate（评估）→ Repair（修复）→ Report（汇报）
```

| 阶段 | 数据采集场景 | 说明 |
|------|-------------|------|
| **Plan** | 用户说"每天上午 10 点采集 4 个视频平台热榜" | 系统解析目标、频率、范围、成功标准，生成结构化任务定义 |
| **Execute** | 定时触发，调用各平台采集脚本 | 并行/串行执行采集，输出原始结果 |
| **Evaluate** | 检查条数是否达标、字段是否完整、时间是否新鲜 | 质量校验规则判断结果是否合格 |
| **Repair** | 某平台失败则自动重试，数据缺失则自动补采 | 自愈策略：重试 → 补采 → 降级 → 人工 |
| **Report** | 推送采集摘要到飞书，含成功率、数据预览、关键异常；遇到无法自愈的异常时主动通知人工介入 | 让用户一眼知道"今天采得怎么样"，关键问题不遗漏 |

### 3.2 分层架构思路

DataClaw 采用"**通用平台内核 + 场景插件**"的分层架构：

```
┌─────────────────────────────────────────────────────────┐
│            交互层（飞书机器人 + 飞书多维表格）               │
│   Skill 触发 / 自然语言输入 / 命令入口 / 结果通知 / 状态查询   │
│              飞书多维表格用于任务列表可视化查看               │
├─────────────────────────────────────────────────────────┤
│                    目标管理层（Goal Parser）               │
│         解析用户需求 → 结构化目标定义 → 任务配置生成           │
├─────────────────────────────────────────────────────────┤
│                  工作流编排层（Workflow Engine）            │
│      任务生命周期 / 定时调度 / 状态机 / 重试策略 / 子任务编排   │
├─────────────────────────────────────────────────────────┤
│                  可插拔执行层（Skills / Plugins）           │
│    腾讯视频采集 / 爱奇艺采集 / 结果校验 / 聚合输出 / 通知发送    │
├─────────────────────────────────────────────────────────┤
│                  评估与决策层（Evaluator & Decision）        │
│         质量校验 / 异常识别 / 自愈决策 / 人工升级判断          │
├─────────────────────────────────────────────────────────┤
│                状态存储与可观测层（State Store）             │
│   任务配置 / 执行记录 / 校验结果 / 补采历史 / 通知日志 / 指标   │
└─────────────────────────────────────────────────────────┘
```

**分层原则**：平台层负责"闭环能力"，插件层负责"具体业务"。新增一个采集场景，只需开发对应的 Skill，无需改动平台内核。

### 3.3 角色划分建议

| 角色          | 承担方                               | 职责                         |
| ----------- | --------------------------------- | -------------------------- |
| **交互与编排中枢** | DataClaw 平台（可基于 Hermes Agent 或自研） | 任务管理、调度触发、状态追踪、通知推送、用户查询响应 |
| **采集技能封装**  | 业务方开发 Skill                       | 采集规范、执行入口、验收标准、常见问题处理      |
| **真实采集执行**  | 业务方脚本 / 服务                        | UI 自动化、接口请求、字段解析、结果标准化     |
| **结果存储与校验** | 平台 + 业务规则                         | 数据落盘、质量判断、补采策略、结果归档        |

**关键原则**：编排层做"任务调度与状态管理"，业务层做"真正去采、真正去解析、真正去校验"。两层解耦，平台才稳。

## 4. 技术拆解

### 4.1 交互层

**目标**：统一承接用户输入和输出，降低使用门槛；支持双模式入口和任务可视化。

**首期形态**：
- **飞书机器人**：主交互入口，支持 Skill 触发、自然语言创建任务、状态查询、通知接收
- **飞书多维表格**：任务可视化入口，集中展示任务列表、当前状态、最近执行结果，支持点击查看详情
- **CLI**：辅助入口，供开发和运维场景使用

**用户能力**：
- **Skill 快速触发**：`/run tencent-hotlist` 一键激活预置采集任务
- **创建自定义任务**：发送完整任务描述，系统解析并生成任务配置；或选择预置模板后填写参数
- **查询任务**："当前有哪些活跃任务？" / 飞书多维表格直接浏览
- **查看结果**："这个任务最近 7 天执行情况"
- **控制任务**：暂停 / 恢复 / 立即执行 / 删除
- **接收通知**：每次执行完成后推送结果摘要（成功率、数据量、关键异常）；失败或需人工介入时即时告警

**交互设计原则**：
- **对话仅作为操作入口**：真正的任务状态、配置、策略全部持久化到系统侧
- **关键操作结构化确认**：创建、修改、删除等操作生成摘要卡片，要求用户确认
- **状态不依赖对话上下文**：新会话中通过任务 ID、名称或多维表格直接查询，不依赖聊天记录
- **多维表格作为状态看板**：解决"不知道有哪些任务在跑"的问题，飞书内即可可视化查看

### 4.2 目标管理层

**目标**：将用户需求解析为结构化的任务定义，支持预置 Skill 直接绑定和自定义任务创建两种模式。

**两种任务创建模式**：

| 模式 | 适用场景 | 输入方式 | 系统行为 |
|------|---------|---------|---------|
| **预置 Skill 绑定** | 常见评测采集场景（榜单、热榜、评论等） | 飞书 Skill 触发（如 `/run tencent-hotlist`）或从列表选择 | 直接加载 Skill 默认配置，用户可调整频率、通知策略等参数后激活 |
| **自定义任务创建** | 复杂、一次性或尚无预置 Skill 的采集需求 | 发送完整任务描述（目标、范围、频率、成功标准、失败处理等） | 系统解析描述 → 提取关键字段 → 生成结构化配置 → 用户确认后创建任务；高频使用的自定义任务可一键沉淀为新 Skill |

**建议抽象字段**（适用于自定义任务创建）：

| 字段 | 说明 | 示例 |
|------|------|------|
| `goal` | 采集目标 | "采集腾讯视频、爱奇艺、优酷、芒果 TV 的热榜数据" |
| `scope` | 采集范围 | 平台列表、榜单类型、字段要求 |
| `constraints` | 约束条件 | 频率（每天 10:00）、超时时间、资源限制 |
| `success_metrics` | 成功标准 | 4 个平台全部成功、字段完整率 100%、数据时效性 < 1 小时 |
| `schedule` | 触发方式 | cron 表达式、事件触发 |
| `failure_policy` | 失败处理 | 自动重试 2 次 → 自动补采 1 次 → 通知人工 |
| `notification_policy` | 通知策略 | 成功发摘要、失败立即告警、升级发加急 |
| `human_review_policy` | 人工介入规则 | 连续 3 次失败、补采仍不完整、未知异常 |

**Skill 配置字段**（预置 Skill 自带，用户可覆盖）：

| 字段 | 说明 |
|------|------|
| `skill_id` | 绑定的 Skill 标识 |
| `skill_params` | Skill 运行参数（如平台列表、榜单类型、返回条数） |
| `skill_defaults` | Skill 默认配置（超时、重试、校验规则） |

### 4.3 工作流编排层

**目标**：负责任务全生命周期管理和闭环执行编排。

**核心能力**：

| 模块 | 功能 | 说明 |
|------|------|------|
| **任务管理** | 增删改查、版本控制、配置快照 | 任务变更可追溯，支持回滚；预置 Skill 任务与自定义任务统一管理 |
| **定时调度** | cron 调度、事件触发、依赖编排 | 支持多任务依赖关系（如"A 完成后触发 B"） |
| **状态机** | 任务状态流转控制 | Draft → Active → Running → Succeeded / Failed / Paused |
| **重试策略** | 失败自动重试、退避策略 | 指数退避、最大重试次数、重试条件判断 |
| **子任务编排** | 多步骤工作流、并行/串行控制 | 四平台可并行采集，校验和聚合串行执行 |
| **日志追踪** | 执行日志、链路追踪 | 每个任务实例完整记录执行过程 |

**任务状态定义**：

```
Draft（草稿） → Active（活跃） → Running（执行中）
                                    ↓
                    ┌────────→ Succeeded（成功）
                    ├────────→ Partially Succeeded（部分成功）
                    ├────────→ Failed（失败）
                    ├────────→ Retrying（重试中）
                    ├────────→ Backfilling（补采中）
                    └────────→ Escalated（已升级人工）
                                    ↓
                              Paused（暂停） / Archived（归档）
```

### 4.4 可插拔执行层（Skills）

**目标**：承载具体业务采集动作，以插件化方式接入；预置 Skill 覆盖常见评测场景，支持自定义 Skill 沉淀。

**Skill 设计原则**：
1. **单一职责**：一个 Skill 只负责一个明确能力单元
2. **输入输出标准化**：统一参数格式和结果 Schema，方便平台调度
3. **执行与说明分离**：业务说明、使用边界、常见问题与执行脚本分离
4. **显式校验与状态返回**：不能只返回"执行了"，还要返回"结果是否可信"
5. **可复用、可组合**：多个 Skill 可被不同场景组合调用
6. **可沉淀**：高频自定义任务可一键导出为新 Skill，供团队复用

**Skill 与任务的关系**：

| 类型 | 说明 | 示例 |
|------|------|------|
| **预置 Skill** | 平台内置，开箱即用，覆盖常见评测场景 | 榜单采集、热榜采集、评论采集 |
| **自定义 Skill** | 从高频自定义任务沉淀而来，团队共享 | 某竞品专属功能采集、特定格式报告生成 |

**首批预置 Skill 示例**（竞品评测 MVP）：

| Skill 名称 | 职责 |
|-----------|------|
| `tencent-video-hotlist` | 腾讯视频热榜采集 |
| `iqiyi-hotlist` | 爱奇艺热榜采集 |
| `youku-hotlist` | 优酷热榜采集 |
| `mgtv-hotlist` | 芒果 TV 热榜采集 |
| `hotlist-validator` | 结果校验：条数、字段完整性、时效性 |
| `hotlist-aggregator` | 多平台结果聚合与摘要生成 |
| `notification-sender` | 飞书消息推送 |

**Skill 内部结构建议**：

```
skill-name/
├── SKILL.md          # 技能说明：能力描述、输入输出、使用边界、常见问题
├── scripts/          # 执行脚本：真正调用采集程序
│   ├── run.py
│   └── validate.py
└── config.yaml       # 默认配置：超时、重试、参数模板
```

### 4.5 评估与决策层

**目标**：让系统具备判断质量、识别异常、决策下一步的能力。

**核心能力**：

| 能力 | 数据采集关注点 |
|------|---------------|
| **状态评估** | 采集是否成功返回 |
| **质量校验** | 条数是否达标、字段是否完整、格式是否正确 |
| **时效校验** | 数据时间是否新鲜、是否过期 |
| **异常识别** | 空值异常、格式异常、接口超时、反爬拦截 |
| **自愈决策** | 是否重试、是否补采、是否降级、是否升级人工 |
| **人工升级** | 连续失败阈值、未知异常类型、高风险场景 |

**自愈策略链路**：

```
执行失败 / 结果不合格
    ↓
判断失败类型（可重试 / 需补采 / 未知异常）
    ↓
可重试 → 自动重试（最多 N 次，指数退避）
    ↓
重试成功 → 标记 Partially Succeeded，记录重试次数
重试失败 → 进入补采判断
    ↓
可补采 → 触发单平台补采（延迟 M 分钟后执行）
    ↓
补采成功 → 标记 Succeeded
补采失败 → 判断是否需要升级人工
    ↓
达到升级阈值 → 标记 Escalated，推送加急告警通知人工
未达阈值 → 标记 Partially Succeeded，记录原因，推送结果摘要
```

**报告与通知策略**：

| 场景 | 通知内容 | 通知方式 |
|------|---------|---------|
| 执行成功 | 结果摘要：成功率、数据量、耗时、关键指标 | 飞书消息 + 多维表格状态更新 |
| 部分成功（自愈后） | 结果摘要 + 自愈记录：哪些平台失败、重试/补采次数、当前状态 | 飞书消息 + 多维表格状态更新 |
| 执行失败（可自愈） | 失败原因、自愈策略、预计恢复时间 | 飞书消息 |
| 无法自愈（需人工） | 升级告警：失败详情、已尝试的恢复动作、建议人工介入点 | 飞书加急消息 |

### 4.6 状态存储与可观测层

**目标**：完整记录任务与执行状态，支撑飞书多维表格可视化、查询、审计和复盘。

**存储内容**：

| 数据类型 | 内容 | 用途 |
|---------|------|------|
| **任务定义** | 任务配置、版本快照、创建/修改历史、绑定的 Skill | 配置管理、变更追溯 |
| **执行记录** | 每次执行实例、起止时间、耗时、执行节点 | 性能分析、故障排查 |
| **原始结果** | 采集原始数据文件路径 / 对象存储地址 | 数据溯源、人工复核 |
| **校验结果** | 质量检查通过/不通过详情、异常项列表 | 质量报告、问题定位 |
| **补采记录** | 补采触发原因、执行结果、重试历史 | 自愈效果评估 |
| **通知记录** | 通知内容、发送时间、送达状态 | 通知审计 |
| **人工介入记录** | 升级原因、处理人、处理结果、耗时 | 人机协作效率分析 |
| **指标曲线** | 任务成功率、数据完整率、自动恢复率、人工介入率 | 平台健康度评估 |

**多维表格数据同步**：
- 任务列表（任务名称、状态、下次执行时间、最近执行结果）
- 执行历史（执行时间、耗时、成功率、数据量、异常摘要）
- 实时状态变更推送至多维表格，飞书内即可查看全貌

**技术选型建议**：
- 任务配置与状态：结构化存储（JSON / SQLite / PostgreSQL）
- 执行日志：可检索日志（本地文件 + 可选集中式日志）
- 原始结果：文件系统 / 对象存储（MinIO / S3）
- 指标数据：时序数据库或关系型数据库的指标表
- 多维表格同步：通过飞书 OpenAPI 定期推送或 webhook 实时更新

## 5. 技术难点

### 5.1 任务创建的双模式平衡

**难点**：预置 Skill 要求覆盖面广、开箱即用，但评测场景多样且变化快；自定义任务创建要求解析准确，但用户描述往往模糊或不完整。两种模式如何平滑切换、互不干扰是关键挑战。

**应对策略**：
- **预置 Skill 优先**：常见评测场景（榜单、热榜、评论、价格）优先沉淀为预置 Skill，用户一键触发即可使用，降低解析压力
- **自定义任务采用"模板 + LLM 解析 + 结构化确认"**：提供常见任务模板供用户选择或参考，LLM 负责理解完整描述，生成结构化配置后向用户展示摘要卡片要求确认
- **建立反馈闭环**：用户修改确认后的自定义配置作为正样本，高频使用的自定义任务可一键沉淀为新 Skill，持续扩充预置库
- **无法解析时降级为引导式问答**：对模糊需求不强行猜测，而是逐步追问澄清关键字段（目标、范围、频率、成功标准）

### 5.2 自循环稳定性与异常边界控制

**难点**：系统在错误方向上持续循环（如重试永远不会成功，却不断触发），或异常场景未被覆盖导致状态卡死。

**应对策略**：
- **Guardrail 机制**：单任务最大重试次数、最大补采次数、最大执行时长、最大失败比例
- **状态机严格约束**：所有状态流转必须明确定义触发条件和允许的目标状态，禁止非法跳转
- **死循环检测**：同一任务实例在短时间内（如 5 分钟）触发超过阈值次数，强制标记 Escalated
- **未知异常熔断**：无法归类的异常不进入默认重试，直接升级人工

### 5.3 评估与归因准确性

**难点**：错误评估或错误归因会导致后续动作偏离目标。例如将"接口被限流"误判为"网络超时"，导致错误的重试策略。

**应对策略**：
- **规则 + LLM 混合评估**：基础质量指标（条数、字段、格式）用确定性规则判断；复杂异常归类和根因分析可引入 LLM，但保留人工纠偏入口
- **归因版本化**：每次评估和归因结果保留版本记录和决策依据，支持事后审计
- **高风险场景人工确认**：对首次出现的异常类型、连续失败的场景，优先要求人工确认归因后再决定策略

### 5.4 多入口状态一致性

**难点**：飞书机器人、飞书多维表格、CLI 三个入口同时操作同一任务时，状态同步和并发控制容易出问题。用户在多维表格中看到一个状态，机器人回复却是另一个状态。

**应对策略**：
- **单一状态源**：所有入口的状态查询都走同一存储层，不缓存副本
- **实时同步**：状态变更通过 webhook 或轮询机制同步到多维表格，确保飞书内查看的数据与系统实际状态一致
- **操作幂等**：关键操作（暂停、恢复、删除）支持幂等执行，避免重复触发
- **冲突检测**：同一任务被多个入口同时操作时，后发起的操作收到冲突提示，需确认后执行

### 5.5 人机协作成本控制

**难点**：人工介入过多会削弱自动化价值；介入过少可能导致问题未被及时发现。

**应对策略**：
- **明确人工介入场景清单**：只有以下场景才升级人工——（1）连续 N 次自愈失败 （2）未知异常类型 （3）数据质量严重不达标且无法自动修复 （4）用户显式配置必须人工确认的策略节点
- **提升通知的可操作性**：告警消息附带"一键确认"、"查看详情"、"跳过本次"等快捷操作按钮，减少人工处理耗时
- **人工介入效果追踪**：记录每次人工介入的问题类型、处理时长、处理结果，持续优化自动化覆盖率

### 5.6 通用性与场景差异化的平衡

**难点**：平台追求通用抽象，但不同采集场景（榜单采集、价格监控、舆情聚合）差异很大，容易抽象过度或过早僵化。

**应对策略**：
- **平台只抽象共性能力**：调度、状态机、通知、存储、人机协作是共性，抽象为平台内核
- **业务差异通过插件承接**：采集逻辑、校验规则、成功标准、修复策略由 Skill 自行定义
- **从具体场景验证抽象**：先用 2–3 个高相似场景验证抽象有效性，再扩展更多场景，避免一开始就追求过度通用

### 5.7 调度与执行环境的稳定性

**难点**：如果基于现有 Agent 框架（如 Hermes Agent）的 cron / skill 能力，需面对其已知限制：cron session 与主进程环境不一致可能导致 skill 依赖不完整；任务输出可能成功但回传信号丢失。

**应对策略**：
- **保守路线起步**：优先让 skill 调用本地 CLI / Python 脚本执行，不将复杂 MCP / toolchain 压入 cron session
- **结果以文件和状态存储为准**：即使通知回传失败，也能通过任务状态查询（飞书机器人或多维表格）和结果文件确认采集是否完成
- **聊天式操作与稳定调度解耦**：cron 负责"生产结果"，飞书负责"查状态和下发新动作"，不依赖"回复某条 cron 消息"的上下文

## 6. 阶段规划

| 阶段                 | 目标                                                    | 关键产出                                     |
| ------------------ | ----------------------------------------------------- | ---------------------------------------- |
| **M3-DC1：预置 Skill 验证** | 验证"飞书 Skill 触发 → 执行 → 结果落盘 → 通知回传"链路打通                | 2–3 个预置采集 Skill（如榜单采集）可一键触发并收到结果摘要       |
| **M3-DC2：闭环能力**        | 完成预置 Skill 的完整闭环：定时调度、结果校验、自动补采、多维表格可视化、主动通知          | V1 可用版本，评测团队可通过 Skill 触发完成日常采集，飞书内查看任务状态 |
| **M3-DC3：自定义任务**       | 支持自定义任务创建与解析：完整任务描述 → 结构化配置 → 任务生成；高频自定义任务可沉淀为新 Skill | 平台支持"预置 + 自定义"双模式，评测团队可灵活应对非标采集需求        |
| **M3-DC4：平台化**         | 统一校验策略、通用 dashboard、更多采集插件、多任务依赖编排                    | 平台级能力沉淀，支持快速扩展新评测场景                      |
| **M3-DC5：复杂场景**        | 从"榜单采集"扩展到"页面级、内容级、行为级采集"                             | 更复杂的采集链路支持                               |
